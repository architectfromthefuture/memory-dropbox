# Memory Substrate Application Areas

Memory Dropbox is an event-first memory substrate: it captures raw items, preserves provenance through events, indexes content for retrieval, and exposes the resulting memory through inspectable APIs and a lightweight UI. The same substrate can support many domain-specific workflows once the capture experience, document extractors, metadata model, and search views are shaped around a target use case.

The larger opportunity is to let intelligence emerge from the substrate. As more observations, files, events, and derived memories accumulate, the system can help detect repeated patterns, weak signals, contradictions, clusters, timelines, and relationships that are hard for humans to hold in working memory. The model input is not just stored content; it becomes a structured field of evidence that can be compared, searched, and revisited from many angles.

This note collects possible application directions. It is intentionally product-facing: each area describes what the substrate could help organize, what additional document types or metadata would matter, and what guardrails should be considered before production use.

## 1. Crime scene intelligence and artifacts

Crime scene intelligence is a strong fit for an inspectable memory substrate because investigations produce fragmented, time-sensitive, multi-format artifacts that need provenance, searchability, and traceability. The goal is not to replace investigators or analysts; it is to create a case memory that can surface overlooked connections, recurring patterns, anomalies, and investigative leads for human review.

### Potential workflow

1. Ingest case artifacts from notes, reports, scene logs, interview transcripts, lab summaries, images, and structured evidence exports.
2. Extract text and metadata into memory items while preserving the original filename, source system, upload timestamp, uploader, case identifier, and chain-of-custody references.
3. Index extracted content for keyword, semantic, and hybrid search across a case workspace.
4. Emit events for each ingestion, extraction, enrichment, review, correction, and search-relevant update.
5. Run pattern-oriented passes over the case memory to surface clusters, repeated descriptions, timeline gaps, conflicting statements, recurring names, unusual locations, and artifacts that resemble prior cases.
6. Present investigators with case-centric views: artifact timeline, people/places/objects index, unanswered questions, linked evidence clusters, pattern alerts, and retrieval traces.

### Useful artifact/document types

- Scene notes and field reports (`.txt`, `.md`, `.docx`, PDF reports)
- CAD/dispatch logs and call narratives
- Evidence inventory exports (`.csv`, `.xlsx`, JSON)
- Chain-of-custody forms and transfer logs
- Witness/victim/officer interview transcripts
- Lab request and result summaries
- Medical examiner or coroner summaries, where legally permitted
- Photos, scanned sketches, diagrams, and maps with OCR/vision-derived descriptions
- Body-worn camera or CCTV transcripts generated by an external transcription pipeline
- Geo/time metadata from authorized devices or evidence systems

### UI enhancements for this direction

- Replace the generic "Upload (text-like)" panel with a case-aware ingestion panel:
  - Case ID / incident ID
  - Artifact type
  - Evidence number
  - Source agency or system
  - Location, observed time, collected time
  - Custodian / uploader
  - Tags such as `weapon`, `vehicle`, `witness`, `timeline`, `lab`, `photo`
- Add an "artifact queue" view that shows extraction status: accepted, extracting, indexed, failed, needs review.
- Add a "case memory" page with filters for artifact type, date/time, person/place/object tags, and ingestion event status.
- Add trace panels that show why a search result appeared: source item, extraction method, indexing job, and related events.
- Add reviewer controls for correcting extracted text and marking derived observations as confirmed, disputed, or irrelevant.
- Add a "patterns humans may miss" panel with human-reviewable suggestions:
  - Similar descriptions across different reports
  - Repeated names, vehicles, objects, phrases, or locations
  - Timeline inconsistencies and unexplained gaps
  - Evidence clusters that share tags, entities, geography, or timestamps
  - Outlier artifacts that do not match the dominant case pattern
  - Cross-case similarities when legally and operationally permitted

### Backend/extraction work required

- Add extractor interfaces for PDF, DOCX, CSV/XLSX, image OCR, and media transcript imports.
- Store original artifact references separately from extracted memory items so the system can distinguish source evidence from derived searchable text.
- Add case-level metadata fields or a typed metadata convention for `case_id`, `artifact_type`, `evidence_id`, `collected_at`, `location`, and `custody_reference`.
- Add ingestion statuses beyond the current text-only path, including extraction failures and reviewer-required states.
- Keep event history append-only so every transformation from artifact to extracted memory is auditable.
- Add derived-memory jobs for pattern candidates, anomaly candidates, entity co-occurrence, timeline reconstruction, and cross-artifact similarity.
- Store model prompts, retrieval context, confidence scores, and review outcomes for every generated lead so analysts can audit how a suggestion emerged.

### Guardrails

- Do not present generated summaries or entity links as facts without human review.
- Treat detected patterns as leads or hypotheses, never determinations.
- Preserve original artifacts and extraction logs; never overwrite source evidence.
- Treat access control, audit logging, encryption, retention, and legal discovery requirements as core product requirements, not later polish.
- Keep chain-of-custody fields explicit and immutable unless corrected through a recorded event.
- Make uncertainty visible in the UI for OCR, transcription, entity extraction, and semantic search results.

## Emergent intelligence patterns

Across all application areas, the substrate becomes more useful when it can identify patterns that are not obvious from any single document.

- **Similarity:** Which artifacts describe the same person, place, object, incident, defect, or claim using different language?
- **Repetition:** Which names, phrases, behaviors, errors, or observations recur across sources?
- **Co-occurrence:** Which entities repeatedly appear together in time, location, tags, or source documents?
- **Timeline reasoning:** What changed, what happened out of order, and where are the gaps?
- **Contradiction:** Which documents make claims that disagree with each other or with structured metadata?
- **Anomaly detection:** Which item does not fit the rest of the case, matter, project, or incident pattern?
- **Weak-signal surfacing:** Which low-frequency details become meaningful when combined with other low-frequency details?
- **Review feedback loops:** Which suggestions did humans confirm, reject, or correct, and how should that affect future surfacing?

These capabilities should remain inspectable. Every surfaced pattern should link back to source artifacts, extracted text, events, model inputs, and review decisions so users can understand why the system raised it.

## 2. Research and knowledge operations

Researchers can use the substrate to collect papers, notes, datasets, meeting transcripts, and web captures into a searchable project memory.

- Document types: Markdown, PDFs, bibliographies, slide decks, transcripts, web exports, CSV datasets.
- UI focus: source credibility, citation trails, project tags, open questions, reading lists, and synthesis notes.
- Added value: hybrid retrieval over a research corpus with visible provenance for each claim or summary.

## 3. Engineering incident memory

Teams can ingest incident timelines, logs, postmortems, Slack exports, deploy notes, and runbooks to build a durable operational memory.

- Document types: postmortem docs, log excerpts, alert exports, deploy metadata, ticket exports, runbooks.
- UI focus: service, severity, incident ID, timeline, contributing factors, mitigations, and follow-up tasks.
- Added value: searchable incident history and traceable lessons learned.

## 4. Legal and compliance matter memory

The substrate can organize matter-specific documents and review notes while preserving event history and retrieval traces.

- Document types: contracts, correspondence, deposition transcripts, evidence logs, regulatory guidance, review memos.
- UI focus: matter ID, privilege/confidentiality labels, document type, custodian, dates, and review status.
- Added value: inspectable search and review workflows where provenance and audit history are essential.

## 5. Healthcare or clinical operations memory

With strict privacy controls, the substrate pattern could help organize operational documents, care coordination notes, policies, and de-identified research material.

- Document types: policies, care protocols, de-identified notes, incident reports, training material.
- UI focus: patient-safety taxonomy, department, policy version, review state, and retention rules.
- Added value: retrieval across operational knowledge while preserving source context and review trails.

## 6. Creative production memory

Creative teams can use the substrate to preserve scripts, briefs, shot lists, image references, feedback, and production decisions.

- Document types: scripts, briefs, image descriptions, transcripts, schedules, feedback docs.
- UI focus: project, scene, character, asset, review status, and version lineage.
- Added value: searchable continuity memory across fast-moving creative work.

## Cross-cutting product themes

- **Typed ingestion:** move from "text-like upload" to selectable document/artifact categories with metadata forms.
- **Extractor pipeline:** add dedicated parsers and OCR/transcription connectors before indexing.
- **Review states:** distinguish raw source material, machine extraction, derived observations, and human-confirmed knowledge.
- **Pattern engines:** run repeatable jobs for similarity, co-occurrence, contradiction, anomaly, and weak-signal detection.
- **Traceable retrieval:** expose source item, ingestion event, extraction method, indexing job, and derived memory links for every result.
- **Workspace boundaries:** add project/case/matter namespaces before supporting sensitive or multi-team use cases.
- **Governance:** design access control, retention, audit logs, and export workflows around the highest-risk target domain.
